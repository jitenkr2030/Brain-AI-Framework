## ğŸ“ Pull Request Process

1. **Ensure tests pass:** All existing and new tests must pass
2. **Update documentation:** Include relevant documentation updates
3. **Clear description:** Explain what your PR does and why
4. **Code standards:** Follow existing code style and patterns
5. **Breaking changes:** Discuss major changes first via issue

### PR Template
```markdown
## Description
Brief description of changes

## Type of Change
- [ ] Bug fix
- [ ] New feature
- [ ] Documentation update
- [ ] Performance improvement
- [ ] Other: ___________

## Testing
- [ ] Tests pass locally
- [ ] Added tests for new functionality
- [ ] Manual testing completed

## Documentation
- [ ] Updated relevant documentation
- [ ] Added examples if applicable
- [ ] Updated CHANGELOG.md

## Checklist
- [ ] Code follows style guidelines
- [ ] Self-review completed
- [ ] Commits are atomic and descriptive
```

## ğŸ› Reporting Issues

### Bug Reports
Include:
- Clear description of the problem
- Steps to reproduce
- Expected vs actual behavior
- Environment details (OS, Python version, etc.)
- Error messages and logs

### Feature Requests
Include:
- Clear problem statement
- Proposed solution
- Alternative solutions considered
- Impact assessment

### Security Issues
**Email:** security@brain-ai.com
- Do not create public issues for security vulnerabilities
- Include detailed description and reproduction steps
- Allow time for security review before public disclosure
